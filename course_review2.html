Problem Definition


(1) Layer: Application Software vs System Software
Middleware: 언어, 에디터, 컴파일러, 데이터베이스, 네트워킹 소프트웨어 등 개발에 필요한 도구
OS-UI, service(kernel), device driver(OS의 지시 받아 추가로 연결된 하드웨어 통제)
(Hypervisor: 가상화, optional)
Firmware(BIOS, Basic Input/Output System): OS 부팅하기 전에 하드웨어 점검, 하드웨어 구동을 위한 최소한의 소프트웨어 실행

(2) Purpose
: own purpose vs request(customer requirement analysis)
SI(Software Integration): 고객의 요구에 따라 SW, HW, 네트워크 등 IT 시스템 통합하여 제공

(3) Legacy
from scratch vs based on existing platform(source code, library)
Reverse Engineering
: 소스 코드 없이 실행 파일만을 가지고 운영하던 기업이 성능 개선, 확장 필요
-> 프로그램 실행 파일로부터 설계도, 요구 사항 등을 추출하여 역개발
Software Reuse
: 라이브러리, COTS, 디자인 패턴 등
COTS: 자체 제작한 특수한 하드웨어는 특수한 소프트웨어 요구
->Arduino, Raspberry Pi 등 상용 하드웨어 사용하면 Software Reuse 가능 디자인 패턴: MVC, MVP 등
+) Design Pattern: MVC, MVP
MVC
: controller에 액션 전달 -> controller에서 model 업데이트 -> controller에서 view 선택
가장 단순, model-view 의존성 높음
MVP
: view로 액션 전달 -> view-presenter-model 로 데이터 요청 -> model-presenter-view 로 데이터 응답
model-view 의존성 없음, view-presenter 의존성 높음

Refactoring
: 코드의 성능, 구조, 가독성 등을 개선
① Readability: 클래스, 변수, 메소드 등 이름, 주석 달기
② Complexity: 함수화
③ Maintainability: 파일 분할, github에 수정 사항 업로드
④ Extensibility: 다른 목적으로 확장하여 사용할 수 있도록 개량
⑤ Performance: 알고리즘
⑥ Fast execution

(4) Project Characteristics
KPI(Key Performance Index/Indicator): SW 비교하는 정량적 지표
ex) lines of code, page load time,
MTBF(Mean Time Between Failure, 길수록 좋음),
MTTR(Mean Time To Repair, 짧을수록 좋음)

(5) Hardware Dependency
Hardware Dependent: 하드웨어가 정해지면 CPU, OS, 언어 등이 결정됨
① Arduino: OS 없음, CPU-ATmega, 언어-C
② Raspberry Pi: CPU-ARM, 입출력-GPIO, USB
③ CUDA: NVIDIA General Purpose GPU(GPGPU) 알고리즘, 언어-C/C++/Fortran
Mobile App Development
: ① Native Platform: iOS-Objective C/Swift, Android-Java/Kotlin
② Cross Platform(one source multi platform): Javascript/React Native, Dart/Flutter
FFI(Foreign Function Interface): 서로 다른 두 언어의 소스 코드를 각각 컴파일 후 하나의 실행 파일로 합침(링크)
